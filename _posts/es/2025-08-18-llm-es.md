Al optimizar el uso de tokens, es recomendable comenzar con modelos más económicos. Si surgen problemas, considera actualizar a modelos más avanzados. Mistral, Gemini Flash y DeepSeek suelen ser económicos, mientras que Claude Sonnet generalmente es más caro. Es crucial entender cómo Claude Code utiliza los routers mostrados a continuación.

En mi experiencia reciente, incurrí en costos significativos por ignorar este principio. Intentaba alcanzar mi límite máximo de uso para determinar el costo, lo cual no es racional; es un cálculo simple. Por ejemplo, ¿realmente necesito Sonnet 4? No necesariamente. Aunque lo percibo como un modelo más avanzado de Anthropic y está altamente clasificado en OpenRouter, no tengo claras las diferencias entre Sonnet 4 y Sonnet 3.5.

Aprendí algo valioso de una [entrevista](https://www.vanta.com/resources/replit-future-of-code) reciente con el fundador de Replit, Amjad Masad. Muchas tareas no requieren modelos altamente avanzados. Idealmente, si podemos evitar usar la API de LLM por completo, sería perfecto. Algunas bibliotecas de NLP son efectivas para tareas simples; por ejemplo, [HanLP](https://github.com/hankcs/HanLP) es excelente para tareas en chino.

Además, podemos desarrollar agentes personalizados o especializados para manejar tareas eficientemente desde el principio. Claude Code no siempre es la mejor o más económica solución para cada tarea.

Una forma de entender las diferencias entre modelos es usarlos extensamente y comparar su rendimiento. Después de usar Gemini 2.5 Flash, encuentro que es menos capaz que Sonnet 4.

Después de unos días, uso la siguiente configuración para ayudar. El parámetro longContextThreshold es realmente importante. Puedes limpiar periódicamente la consola de Claude Code o reiniciarla. Es muy fácil alcanzar el umbral de contexto largo cuando escribes código con Claude Code.

## Agentes Generales vs Agentes Verticales

*2025.08*

[Manus](https://manus.im) se anuncia como una herramienta de agente de IA general, pero probablemente no funcione tan bien.

Una razón es que es muy lento, realizando mucho trabajo innecesario y siendo ineficiente. Otra razón es que, si encuentra un problema complejo o un punto débil, es probable que falle en tu tarea.

Los agentes verticales funcionan muy bien porque están altamente especializados. Están diseñados para tareas muy específicas. Hay docenas de bases de datos y más de cien frameworks de desarrollo web como Spring. También hay numerosos frameworks web, como Vue o React.

[Dify](https://dify.ai) se centra en usar IA para conectar flujos de trabajo, empleando un método de arrastrar y conectar para definir flujos de IA. Necesitan hacer mucho para conectar información, datos y plataformas.

Yo también he construido algunos agentes simples, como un agente de refactorización de código Python, un agente de corrección de gramática, un agente de corrección de errores y un agente de fusión de ensayos.

El código es muy flexible. Así que Dify solo cubre una pequeña parte del espacio de ideas posibles.

Manus realiza tareas y muestra a los usuarios cómo funciona mediante un método VNC para mostrar una computadora.

Creo que el futuro se definirá entre estos dos enfoques.

Para Manus, necesitas subir código o texto para realizar tareas, lo cual no es conveniente. Con Dify, debes construir flujos de trabajo usando arrastrar y soltar, similar a MIT Scratch.

¿Por qué Scratch no es tan popular como Python? Porque con Python puedes hacer muchas cosas, mientras que Scratch está limitado a programas simples con fines educativos.

Dify probablemente tenga limitaciones similares.

Manus puede manejar muchas tareas simples. Sin embargo, para algunas tareas, especialmente aquellas que tocan sus puntos débiles, fallará.

Además, muchos programas o servicios requieren tiempo para configurarse. En el enfoque de Manus, este proceso es lento.

Como programador, uso IA con Python para construir mis agentes verticales. Este es el enfoque más simple para mí. También puedo configurar prompts y contextos para garantizar una salida relativamente estable de las APIs de LLM.

Manus y Dify también están construidos con estas APIs de LLM. Su ventaja es que ya tienen muchas herramientas o código listo para usar.

Si quiero construir un agente bot para Twitter, usar Dify puede ser más conveniente que construirlo yo mismo con tecnologías de código abierto.

## Opinión de un Ingeniero Exigente sobre Herramientas de IA para Programación

*2025.08*

Recientemente, logré ejecutar Claude Code, así que quiero compartir mi viaje en la selección de herramientas. También he recopilado algunos [Consejos sobre Herramientas de IA](ai-tips-en.md) en el camino.

Fui bastante tardío en adoptar Claude Code.

[Claude Code](https://www.anthropic.com/news/claude-3-5-sonnet) se lanzó a finales de febrero de 2025.

No logré probarlo con éxito hasta hace poco. Una razón es que requiere la API de Anthropic, que no admite tarjetas Visa chinas.

Otra razón es que [Claude Code Router](https://github.com/musistudio/claude-code-router) estuvo disponible, lo que hizo exitoso mi intento reciente.

Sigo escuchando elogios sobre él. Probé la CLI de Gemini en julio de 2025, pero la abandoné después de varios intentos fallidos de que corrigiera mi código.

También probé Aider, otro agente de software. Dejé de usar Cursor después de unos seis meses porque muchos de sus plugins basados en VSCode fallaban. Además, no quiero darle mucho crédito a Cursor, ya que está construido sobre VSCode. Como el plugin Copilot en VSCode ha mejorado recientemente y no se queda muy atrás, prefiero usarlo con más frecuencia.

Sin embargo, VSCode está construido sobre Electron, una tecnología de código abierto. Es difícil atribuir crédito al equipo o individuo correcto. Considerando que muchas empresas y startups se benefician de proyectos de código abierto, debo enfocarme en mi presupuesto y lo que mejor me funciona. No debería preocuparme demasiado por dar crédito. Prefiero usar herramientas asequibles y efectivas.

Experimenté brevemente con Cline, pero no la adopté.

Uso el plugin Copilot en VSCode con un modelo personalizado, Grok 3 beta a través de OpenRouter, y funciona bien.

No creo que Claude Code cambie mis hábitos, pero como puedo ejecutarlo con éxito y tengo paciencia para probarlo algunas veces más, veré cómo me siento en las próximas semanas.

Soy un usuario exigente con 10 años de experiencia en ingeniería de software. Espero que las herramientas sean geniales en el uso real. No me importa la marca, solo la utilidad diaria.

Después de usar Claude Code para corregir la gramática de esta publicación, encontré que funciona bien en ciertos escenarios. Aunque aprecio la IA para asistencia gramatical (incluso escribí un script en Python para llamar a APIs de LLM con este fin), he notado un patrón frustrante: incluso cuando solicito correcciones mínimas, las herramientas siguen mostrando numerosas sugerencias gramaticales para revisar. Este proceso de verificación manual anula el propósito de la automatización. Como compromiso, ahora dejo que la IA maneje ensayos completos, aunque este enfoque limita mis oportunidades de aprendizaje, ya que no veo las correcciones específicas.

Lo que más me impresionó fue cómo Claude Code muestra los cambios: mostrando comparaciones antes y después, similares a los diffs de git, lo que hace que revisar las ediciones sea mucho más fácil.

Después de un día, usé Claude para corregir algo de código también. Sin embargo, sigo usando el plugin Copilot con el modelo Grok 3 beta, ya que es simple y fácil para mí.

Después de usar Claude Code durante varios días, tengo que decir que es muy impresionante. Realmente me gusta cómo corrige mi código.

## Uso de API de Deepseek y Mistral

*2025.01.25*

### DeepSeek

En un mes, 15 millones de tokens me costaron aproximadamente 23.5 CNY.

Este fue mi uso en un día:

| Tipo               | Tokens    |
|--------------------|-----------|
| Entrada (Cache Hit) | 946,816   |
| Entrada (Cache Miss)| 2,753,752 |
| Salida             | 3,100,977 |

El cálculo es el siguiente:

0.94 * 0.1 + 2.75 * 1 + 3.10 * 2 = 11.83

Por lo tanto, dependiendo de la tarea, el uso de tokens depende en gran medida de la entrada (cache miss) y la salida.

Este resultado coincide con el costo esperado.

### Mistral

Los precios de los modelos de Mistral son los siguientes:

| Modelo               | Entrada (USD por millón de tokens) | Salida (USD por millón de tokens) |
|----------------------|-----------------------------------|----------------------------------|
| `mistral-large-2411` | 2                                 | 6                                |
| `mistral-small-latest`| 0.2                              | 0.6                              |

En un día, el uso de mi cuenta en Mistral fue el siguiente (Modelo: `mistral-large-2411`):

| Tipo   | Tokens  | Costo (USD) |
|--------|---------|------------|
| Total  | 772,284 | 3.44       |
| Salida | 474,855 | 2.85       |
| Entrada| 297,429 | 0.59       |

Para el modelo `mistral-small-2409`, el uso total fue de 1,022,407 tokens.

Asumiendo que 1/3 fueron tokens de entrada y 2/3 fueron de salida:

Hubo 340,802 tokens de entrada y 681,605 tokens de salida.

Por lo tanto, el costo total se calcula como 340,802 * 0.2 / 1,000,000 + 681,605 * 0.6 / 1,000,000 = 0.07 + 0.41 = 0.48 USD.

La consola de Mistral reporta un costo total de uso de 0.43 USD, lo cual coincide aproximadamente con nuestro cálculo.
