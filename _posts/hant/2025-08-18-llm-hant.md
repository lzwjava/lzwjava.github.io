在優化令牌使用量時，建議從更具成本效益的模型開始。如果出現問題，請考慮升級到更先進的模型。Mistral、Gemini Flash 和 DeepSeek 通常比較經濟，而 Claude Sonnet 通常更昂貴。了解 Claude Code 如何使用以下顯示的路由器至關重要。

在我最近的經驗中，由於忽視了這項原則，我產生了相當大的成本。我當時正試圖達到我的最大使用量以確定成本，這不是一種理性的方法；這是一個簡單的計算。例如，我真的需要 Sonnet 4 嗎？不一定。儘管我認為它是 Anthropic 更先進的模型，並且在 OpenRouter 上排名很高，但我並不清楚 Sonnet 4 和 Sonnet 3.5 之間的區別。

我從最近對 Replit 創始人 Amjad Masad 的[採訪](https://www.vanta.com/resources/replit-future-of-code)中學到了一些有價值的東西。對於許多任務而言，高度先進的模型並非必要。理想情況下，如果我們能完全避免使用大型語言模型 API，那就再好不過了。某些自然語言處理庫對於更簡單的任務來說是有效的；例如，[HanLP](https://github.com/hankcs/HanLP) 在處理中文任務方面表現出色。

此外，我們可以開發客製化或專門的代理來從一開始就高效處理任務。Claude Code 未必總是每個任務的最佳或最具成本效益的解決方案。

區分模型差異的一種方法是廣泛使用它們並比較它們的性能。使用 Gemini 2.5 Flash 一段時間後，我覺得它的功能不如 Sonnet 4。

## 通用代理與垂直代理

*2025.08*

[Manus](https://manus.im) 聲稱是一個通用的 AI 代理工具，但它可能不會那麼好用。

一個原因是因為它速度很慢，做了很多不必要的工作且效率低下。另一個原因是，如果它遇到複雜問題或碰到弱點，你的任務很可能會失敗。

垂直代理表現出色，因為它們高度專業化。它們專為非常特定的任務量身定制。有數十個資料庫和一百多個網頁開發框架，例如 Spring。還有許多網頁框架，例如 Vue 或 React。

[Dify](https://dify.ai) 專注於利用 AI 連結工作流程，採用拖放連接的方法來定義 AI 工作流程。他們需要做很多工作來連接資訊、數據和平台。

我也建立了一些簡單的代理，例如 Python 程式碼重構代理、語法修正代理、錯誤修正代理和文章合併代理。

程式碼非常靈活。因此，Dify 只涵蓋了所有可能想法的一小部分空間。

Manus 透過使用 VNC 方法顯示電腦來執行任務並向用戶展示其工作原理。

我認為未來將會在這兩種方法上穩定下來。

對於 Manus，你需要上傳程式碼或文字來執行任務，這不方便。對於 Dify，你需要使用拖放建立工作流程，這類似於麻省理工學院的 Scratch。

為什麼 Scratch 不像 Python 那樣受歡迎？因為使用 Python，你可以做很多事情，而 Scratch 僅限於用於教育目的的簡單程式。

Dify 可能也有類似的限制。

Manus 可以處理許多簡單的任務。但是，對於某些任務，特別是那些觸及 Manus 弱點的任務，它將會失敗。

此外，許多程式或服務需要時間設置。在 Manus 的方法中，這個過程很慢。

作為程式設計師，我使用 Python 結合 AI 來建構我的垂直代理。這對我來說是最簡單的方法。我還可以設置提示和上下文，以確保從大型語言模型 API 獲得相對穩定的輸出。

Manus 和 Dify 也是使用這些大型語言模型 API 建構的。它們的優勢在於它們已經有許多工具或程式碼可供使用。

如果我想建立一個 Twitter 機器人代理，使用 Dify 可能比我自己使用開源技術建立一個更方便。

## 一個挑剔工程師對 AI 編碼工具的看法

*2025.08*

最近，我成功運行了 Claude Code，所以我想分享我的工具選擇歷程。我也在此過程中收集了一些 [AI 工具提示](ai-tips-en.md)。

我遲遲才採用 Claude Code。

[Claude Code](https://www.anthropic.com/news/claude-3-5-sonnet) 約於 2025 年 2 月底發布。

直到最近我才成功嘗試。一個原因是它需要 Anthropic API，而該 API 不支援中國銀聯卡。

另一個原因是 [Claude Code Router](https://github.com/musistudio/claude-code-router) 的出現，使我最近的嘗試成功。

我一直聽到對它的讚揚。我在 2025 年 7 月嘗試了 Gemini CLI，但在幾次嘗試修復我的程式碼失敗後放棄了它。

我也試過 Aider，這是另一個軟體代理。我大約在六個月後停止使用 Cursor，因為它的許多基於 VSCode 的外掛程式都失效了。此外，我不想給 Cursor 太多功勞，因為它是基於 VSCode 建立的。由於 VSCode 中的 Copilot 外掛程式最近有所改進，並且沒有遠遠落後，我更喜歡經常使用它。

然而，VSCode 是基於 Electron 這一開源技術建構的。很難將功勞歸於合適的團隊或個人。考慮到許多大公司和新創公司都從開源專案中獲利，我必須專注於我的預算以及什麼最適合我。我不必太擔心歸功。我更喜歡使用經濟實惠且有效的工具。

我曾短暫嘗試過 Cline 但沒有採用。

我在 VSCode 中使用 Copilot 外掛程式，並透過 OpenRouter 使用自訂模型 Grok 3 beta，效果很好。

我不認為 Claude Code 會改變我的習慣，但既然我能成功運行它，並有耐心再試幾次，我將在未來幾週看看我的感受。

我是一個有 10 年軟體工程經驗的挑剔使用者。我希望工具在實際使用中能表現出色。我不相信品牌——我只關心日常實用性。

使用 Claude Code 修正這篇文章的語法後，我發現它在某些情況下表現良好。雖然我感謝 AI 在語法方面的協助（我甚至編寫了一個 Python 腳本來呼叫 LLM API 以達到此目的），但我注意到一個令人沮喪的模式——即使我要求極少的修正，這些工具仍會持續浮現大量語法建議供我審閱。這種手動驗證的過程違背了自動化的目的。作為折衷，我現在讓 AI 處理整篇文章，儘管這種方法限制了我的學習機會，因為我沒有看到正在進行的具體修正。

最讓我印象深刻的是 Claude Code 顯示變更的方式——顯示類似 Git Diff 的前後比較，這讓審閱編輯變得容易得多。

一天之後，我也用 Claude 修復了一些程式碼。然而，我繼續使用帶有 Grok 3 beta 模型的 Copilot 外掛程式，因為它對我來說簡單易用。

使用 Claude Code 幾天後，我不得不說它確實令人印象深刻。我非常喜歡它修復我的程式碼的方式。

## Deepseek 和 Mistral 的 API 使用

*2025.01.25*

### DeepSeek

一個月內，1500 萬個令牌花費了我約 23.5 人民幣。

這是我一天內的使用量：

| 類型              | 令牌數    |
|-------------------|-----------|
| 輸入（快取命中）  | 946,816   |
| 輸入（快取未命中） | 2,753,752 |
| 輸出            | 3,100,977 |

計算方式如下：

0.94 * 0.1 + 2.75 * 1 + 3.10 * 2 = 11.83

因此，視任務而定，令牌使用量主要取決於輸入（快取未命中）和輸出。

這個結果與預期成本一致。

### Mistral

Mistral 模型的定價如下：

| 型號                  | 輸入（每百萬令牌美元） | 輸出（每百萬令牌美元） |
|-----------------------|------------------------------|---------------------------------|
| `mistral-large-2411`  | 2                            | 6                               |
| `mistral-small-latest`| 0.2                          | 0.6                             |

一天內，我的 Mistral 帳戶使用量如下（模型：`mistral-large-2411`）：

| 類型   | 令牌數  | 成本 (美元) |
|--------|---------|------------|
| 總計  | 772,284 | 3.44       |
| 輸出 | 474,855 | 2.85       |
| 輸入  | 297,429 | 0.59       |

對於 `mistral-small-2409` 模型，總使用量為 1,022,407 個令牌。

假設其中 1/3 為輸入令牌，2/3 為輸出令牌：

有 340,802 個輸入令牌和 681,605 個輸出令牌。

因此，總成本計算為 340,802 * 0.2 / 1,000,000 + 681,605 * 0.6 / 1,000,000 = 0.07 + 0.41 = 0.48 美元。

Mistral 控制台報告的總使用成本為 0.43 美元，這與我們的計算大致相符。
