[Manus](https://manus.im) 被宣稱是一個通用 AI 代理工具，但它可能不會運作得很好。

其中一個原因是它非常緩慢，做了很多不必要的工作，效率低下。另一個原因是如果它遇到複雜的問題或觸及弱點，你很可能會在任務上失敗。

自訂代理非常有效，因為它們高度專業化。它們是為非常特定的任務量身定制的。有數十個數據庫和超過一百個網頁開發框架，例如 Spring。還有許多網頁框架，例如 Vue 或 React。

[Dify](https://dify.ai) 專注於使用 AI 連接工作流程，採用拖放連接的方法來定義 AI 工作流程。它們需要做很多工作來連接資訊、數據和平台。

我也建立了一些簡單的代理，例如 Python 代碼重構代理、文法修正代理、錯誤修復代理和文章合併代理。

代碼非常靈活。因此，Dify 只涵蓋了可能想法空間的一小部分。

Manus 通過使用 VNC 方法顯示電腦來執行任務並向用戶展示其運作方式。

我認為未來將會在這兩種方法上取得共識。

對於 Manus，你需要上傳代碼或文字來執行任務，這不太方便。而 Dify 需要通過拖放來建立工作流程，類似於 MIT Scratch。

為什麼 Scratch 不如 Python 受歡迎？因為用 Python 可以做很多事情，而 Scratch 只限於簡單的教育用程式。

Dify 可能有類似的限制。

Manus 可以處理很多簡單的任務。但對於某些任務，特別是觸及 Manus 弱點的任務，它會失敗。

此外，許多程式或服務需要時間來設置。在 Manus 的方法中，這個過程是緩慢的。

作為程式員，我使用 Python 和 AI 來建立我的自訂代理。這是對我來說最簡單的方法。我也可以設置提示和上下文，以確保來自 LLM API 的輸出相對穩定。

Manus 和 Dify 也是用這些 LLM API 建造的。它們的優勢在於它們已經有很多工具或代碼可以直接使用。

如果我想建立一個 Twitter 機器人代理，使用 Dify 可能比自己用開源技術建造更方便。
