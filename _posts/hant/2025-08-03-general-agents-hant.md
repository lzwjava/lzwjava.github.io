[Manus](https://manus.im) 聲稱是一款通用 AI 代理工具，但實際上可能效果並不理想。

原因之一是它運行速度非常慢，會執行大量不必要的工作，效率低下。另一個原因是，當它遇到複雜問題或觸及其弱點時，用戶很可能無法完成任務。

垂直領域的 AI 代理表現更佳，因為它們高度專業化，針對非常具體的任務量身定制。目前有數十種數據庫，以及超過百種網頁開發框架，例如 Spring。此外，還有許多網頁框架，如 Vue 或 React。

[Dify](https://dify.ai) 專注於利用 AI 連接工作流程，採用拖放連接的方式定義 AI 工作流。他們需要做大量工作來整合資訊、數據和平台。

我也構建過一些簡單的 AI 代理，例如 Python 代碼重構代理、語法修正代理、Bug 修復代理和文章合併代理。

代碼非常靈活，因此 Dify 只能覆蓋極小部分的可能應用場景。

Manus 透過 VNC 方法展示電腦操作，在執行任務的同時向用戶展示其工作過程。

我認為未來將會在這兩種模式中確定方向。

使用 Manus 需要上傳代碼或文本才能執行任務，操作上並不方便。而 Dify 則需要透過拖放方式構建工作流，類似於 MIT 的 Scratch。

為何 Scratch 沒有 Python 普及？因為 Python 能完成的事情太多，而 Scratch 只限於教育用途的簡單程式。

Dify 或許也有類似的局限性。

Manus 能處理許多簡單任務，但對於某些任務，特別是觸及其弱點的任務，它就會失敗。

此外，許多程式或服務需要時間設定，而在 Manus 的模式下，這個過程會變得緩慢。

作為程式開發人員，我使用 Python 和 AI 來構建垂直領域的代理。這是最簡單的方式，我也能透過設定提示詞和上下文，確保從 LLM API 得到相對穩定的輸出。

Manus 和 Dify 同樣基於這些 LLM API。它們的優勢在於已經準備好許多可用的工具或代碼。

如果我想構建一個 Twitter 機器人代理，使用 Dify 可能會比自己用開源技術構建更方便。
