[Manus](https://manus.im) 声称是一款通用 AI 代理工具，但实际上可能表现不佳。

原因之一是它运行速度很慢，做了很多无谓的工作且效率低下。另一个原因是，如果遇到复杂问题或触及其薄弱环节，任务很可能会失败。

垂直领域的 AI 代理效果更好，因为它们高度专业化，针对特定任务量身定制。目前有数十种数据库和上百种 Web 开发框架（如 Spring），还有众多 Web 框架（如 Vue 或 React）。

[Dify](https://dify.ai) 专注于用 AI 连接工作流，通过拖拽方式定义 AI 工作流。它们需要做大量工作来连接信息、数据和平台。

我也构建过一些简单的代理，比如 Python 代码重构代理、语法修正代理、Bug 修复代理和文章合并代理。

代码非常灵活，因此 Dify 只覆盖了极小一部分的可能性空间。

Manus 通过 VNC 方式展示电脑操作，执行任务并向用户展示过程。

我认为未来会在这两种方法上形成共识。

使用 Manus 需要上传代码或文本才能执行任务，不太方便。而 Dify 需要通过拖拽构建工作流，类似 MIT Scratch。

为什么 Scratch 不如 Python 流行？因为 Python 能做的事情太多，而 Scratch 仅限于教育用途的简单程序。

Dify 可能也有类似的局限性。

Manus 能处理很多简单任务，但遇到其薄弱环节时，会失败。

此外，许多程序或服务需要时间设置，Manus 的方法会让这个过程变得缓慢。

作为程序员，我用 Python + AI 构建垂直代理，这是最简单的方式。我也能设置提示词和上下文，确保 LLM API 输出相对稳定。

Manus 和 Dify 也是基于这些 LLM API 构建的。它们的优势在于已集成了大量现成工具或代码。

如果我想构建一个 Twitter 机器人代理，用 Dify 可能比自己用开源技术搭建更方便。
