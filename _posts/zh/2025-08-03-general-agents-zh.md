[Manus](https://manus.im) 被宣称是一个通用 AI 代理工具，但它可能不会表现得很好。

其中一个原因是它非常慢，做了很多不必要的工作，效率低下。另一个原因是，如果它遇到复杂问题或触及弱点，你很可能会在任务中失败。

定制化代理表现很好，因为它们高度专业化。它们是为非常特定的任务量身定制的。有数十个数据库和超过一百种网络开发框架，如 Spring。还有许多网络框架，例如 Vue 或 React。

[Dify](https://dify.ai) 专注于使用 AI 连接工作流，采用拖拽连接的方法定义 AI 工作流。它们需要做很多工作来连接信息、数据和平台。

我也构建了一些简单的代理，例如 Python 代码重构代理、语法修复代理、错误修复代理和文章合并代理。

代码非常灵活。因此，Dify 只覆盖了可能想法空间的一小部分。

Manus 通过使用 VNC 方法显示计算机来执行任务并向用户展示其工作方式。

我认为未来将会在这两种方法上达成共识。

对于 Manus，你需要上传代码或文本来执行任务，这并不方便。对于 Dify，你需要使用拖放的方式构建工作流，类似于 MIT Scratch。

为什么 Scratch 不如 Python 受欢迎？因为用 Python 可以做很多事情，而 Scratch 仅限于简单的教育程序。

Dify 可能也有类似的局限性。

Manus 可以处理很多简单任务。然而，对于某些任务，尤其是触及 Manus 弱点的任务，它会失败。

此外，许多程序或服务需要时间来设置。在 Manus 的方法中，这个过程很慢。

作为程序员，我使用 Python 构建 AI 来构建我的定制代理。这是我最简单的方法。我也可以设置提示和上下文，以确保从 LLM API 获得相对稳定的输出。

Manus 和 Dify 也是用这些 LLM API 构建的。它们的优势在于它们已经有很多工具或代码可以直接使用。

如果我想构建一个 Twitter 机器人代理，使用 Dify 可能比自己用开源技术构建更方便。
