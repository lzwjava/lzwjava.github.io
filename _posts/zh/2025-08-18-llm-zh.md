在优化 token 使用时，建议从更具成本效益的模型开始。如果出现问题，可以考虑升级到更高级的模型。Mistral、Gemini Flash 和 DeepSeek 通常经济实惠，而 Claude Sonnet 通常更昂贵。了解 Claude Code 如何使用下图中所示的路由器至关重要。

在我最近的经验中，由于忽视了这一原则，我花费了大量成本。我试图达到我的最大使用量以确定成本，这不是一种理性的方法；这只是一个简单的计算。例如，我真的需要 Sonnet 4 吗？不一定。尽管我将其视为 Anthropic 更高级的模型，并且它在 OpenRouter 上排名很高，但我还不清楚 Sonnet 4 和 Sonnet 3.5 之间的区别。

我从最近对 Replit 创始人 Amjad Masad 的[采访](https://www.vanta.com/resources/replit-future-of-code)中了解到了一些有价值的东西。对于许多任务，并不需要高度先进的模型。理想情况下，如果我们能完全避免使用 LLM API，那就完美了。某些 NLP 库对于更简单的任务很有效；例如，[HanLP](https://github.com/hankcs/HanLP) 在处理中文语言任务方面表现出色。

此外，我们可以开发定制或专门的代理，从一开始就高效地处理任务。Claude Code 可能并不总是处理所有任务的最佳或最具成本效益的解决方案。

识别模型之间差异的一种方法是大量使用它们并比较它们的性能。在使用 Gemini 2.5 Flash 一段时间后，我发现它不如 Sonnet 4。

## 通用代理与垂直代理

*2025.08*

[Manus](https://manus.im) 号称是一个通用的 AI 代理工具，但它可能不会那么好用。

一个原因是它非常慢，做了很多不必要的工作，效率低下。另一个原因是，如果它遇到复杂问题或遇到弱点，你的任务很可能会失败。

垂直代理效果很好，因为它们高度专业化。它们是为非常具体的任务量身定制的。有几十个数据库和一百多个像 Spring 这样的 Web 开发框架。还有许多 Web 框架，如 Vue 或 React。

[Dify](https://dify.ai) 专注于使用 AI 连接工作流，采用拖拽连接的方式定义 AI 工作流。他们需要做很多工作来连接信息、数据和平台。

我也构建了一些简单的代理，例如 Python 代码重构代理，语法修复代理，错误修复代理和文章合并代理。

代码非常灵活。所以，Dify 只覆盖了可能想法的一小部分空间。

Manus 通过使用 VNC 方法显示计算机来执行任务并向用户展示其工作方式。

我认为未来将趋于这两种方法。

对于 Manus，你需要上传代码或文本来执行任务，这不方便。对于 Dify，你需要使用拖拽来构建工作流，类似于 MIT Scratch。

为什么 Scratch 不像 Python 那么受欢迎？因为使用 Python，你可以做很多事情，而 Scratch 仅限于教育用途的简单程序。

Dify 可能也有类似的限制。

Manus 可以处理许多简单任务。但是，对于某些任务，尤其是那些触及 Manus 弱点的任务，它会失败。

此外，许多程序或服务都需要时间来设置。在 Manus 的方法中，这个过程很慢。

作为一名程序员，我使用 Python 结合 AI 来构建我的垂直代理。这是对我来说最简单的方法。我还可以设置提示和上下文，以确保 LLM API 的输出相对稳定。

Manus 和 Dify 也是用这些 LLM API 构建的。它们的优势在于它们已经有很多现成的工具或代码。

如果我想构建一个 Twitter 机器人代理，使用 Dify 可能比我自己用开源技术构建一个更方便。

## 一个挑剔工程师对 AI 编程工具的看法

*2025.08*

最近，我成功运行了 Claude Code，所以我想分享我的工具选择之旅。我还收集了一些 [AI 工具提示](ai-tips-en.md)。

我采用 Claude Code 相当晚。

[Claude Code](https://www.anthropic.com/news/claude-3-5-sonnet) 于 2025 年 2 月底发布。

直到最近我才成功尝试它。一个原因是因为它需要 Anthropic API，而它不支持中国的 Visa 卡。

另一个原因是因为 [Claude Code Router](https://github.com/musistudio/claude-code-router) 变得可用，这使得我最近的尝试成功了。

我一直听到对它的赞扬。我于 2025 年 7 月尝试了 Gemini CLI，但在几次尝试修复我的代码失败后就放弃了。

我还尝试了 Aider，这是另一个软件代理。我在大约六个月后停止使用 Cursor，因为它的许多基于 VSCode 的插件都出了故障。此外，我不想给 Cursor 太多赞誉，因为它是在 VSCode 之上构建的。由于 VSCode 中的 Copilot 插件最近有所改进，并且并未落后太多，我更喜欢经常使用它。

然而，VSCode 是基于 Electron（一种开源技术）构建的。很难将功劳归因于正确的团队或个人。考虑到许多大公司和初创公司都从开源项目中获利，我必须专注于我的预算以及最适合我的东西。我不必太担心归功于谁。我更喜欢使用经济实惠且有效的工具。

我曾简短地尝试过 Cline，但没有采用它。

我使用 VSCode 中的 Copilot 插件，并使用自定义模型 Grok 3 beta，通过 OpenRouter，效果很好。

我认为 Claude Code 不会改变我的习惯，但既然我能成功运行它并有耐心再试几次，我会在接下来的几周看看感觉如何。

我是一个有 10 年软件工程经验的挑剔用户。我希望工具在实际使用中能表现出色。我不相信品牌——我只关心日常的实用性。

在使用 Claude Code 修复这篇文章的语法后，我发现它在某些场景下效果很好。虽然我很感谢 AI 在语法方面的帮助（我甚至为此写了一个 Python 脚本来调用 LLM API），但我注意到一个令人沮丧的模式——即使我要求最少的修复，这些工具也会不断提出大量的语法建议供我审查。这种人工验证过程违背了自动化的目的。作为折衷，我现在让 AI 处理整篇文章，尽管这种方法限制了我的学习机会，因为我无法看到具体的修改。

最让我印象深刻的是 Claude Code 显示更改的方式——它显示了类似 git diff 的前后比较，这使得审查修改变得容易得多。

一天后，我也用 Claude 修复了一些代码。然而，我继续使用带有 Grok 3 beta 模型的 Copilot 插件，因为它对我来说简单易用。

在使用 Claude Code 几天后，我不得不说它非常令人印象深刻。我真的很喜欢它修复我代码的方式。

## Deepseek 和 Mistral 的 API 使用

*2025.01.25*

### DeepSeek

一个月内，1500 万 token 花费了我大约 23.5 元人民币。

这是我一天的使用情况：

| 类型             | token     |
|------------------|-----------|
| 输入（缓存命中） | 946,816   |
| 输入（缓存未命中） | 2,753,752 |
| 输出             | 3,100,977 |

计算如下：

0.94 * 0.1 + 2.75 * 1 + 3.10 * 2 = 11.83

因此，取决于任务，token 使用量主要取决于输入（缓存未命中）和输出。

这个结果与预期成本相符。

### Mistral

Mistral 模型的定价如下：

| 型号                  | 输入（每百万 token 美元） | 输出（每百万 token 美元） |
|-----------------------|-----------------------|-------------------------|
| `mistral-large-2411`  | 2                     | 6                       |
| `mistral-small-latest`| 0.2                   | 0.6                     |

一天内，我的 Mistral 账户使用情况如下（型号：`mistral-large-2411`）：

| 类型   | token   | 成本（美元） |
|--------|---------|------------|
| 总计   | 772,284 | 3.44       |
| 输出   | 474,855 | 2.85       |
| 输入   | 297,429 | 0.59       |

对于 `mistral-small-2409` 模型，总使用量为 1,022,407 token。

假设其中 1/3 为输入 token，2/3 为输出 token：

有 340,802 个输入 token 和 681,605 个输出 token。

因此，总成本计算为 340,802 * 0.2 / 1,000,000 + 681,605 * 0.6 / 1,000,000 = 0.07 + 0.41 = 0.48 美元。

Mistral 控制台报告的总使用成本为 0.43 美元，这与我们的计算近似匹配。
