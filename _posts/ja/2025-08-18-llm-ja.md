トークン利用を最適化する際、費用対効果の高いモデルから始めることをお勧めします。問題が発生した場合は、より高度なモデルへのアップグレードを検討してください。Mistral、Gemini Flash、DeepSeekは通常経済的であり、Claude Sonnetは一般的に高価です。Claude Codeが以下に示すルーターをどのように使用しているかを理解することが重要です。

最近の私の経験では、この原則を無視したため、かなりの費用がかかりました。コストを決定するために最大使用量に到達しようとしていたのですが、これは合理的なアプローチではありません。単なる計算です。たとえば、Sonnet 4が本当に必要でしょうか？必ずしもそうではありません。Anthropicのより高度なモデルであり、OpenRouterで上位にランク付けされていると認識していますが、Sonnet 4とSonnet 3.5の違いについては不明です。

最近、Replitの創設者であるAmjad Masad氏との[インタビュー](https://www.vanta.com/resources/replit-future-of-code)から貴重なことを学びました。そして、多くのタスクにおいて、非常に高度なモデルは必要ありません。理想的には、LLM APIの使用を完全に避けられれば完璧です。特定のNLPライブラリは、より簡単なタスクに効果的です。たとえば、[HanLP](https://github.com/hankcs/HanLP)は中国語のタスクの処理に優れています。

さらに、タスクを効率的に最初から処理するために、カスタムまたは専門のエージェントを開発できます。Claude Codeがすべてのタスクに最適な、または最も費用対効果の高いソリューションであるとは限りません。

モデル間の違いを識別する1つの方法は、それらを広く使用し、パフォーマンスを比較することです。Gemini 2.5 Flashをしばらく使用した後、Sonnet 4よりも能力が低いと感じています。

```json
{
  "Router": {
    "default": "openrouter,google/gemini-2.5-flash",
    "background": "openrouter,google/gemini-2.5-flash",
    "think": "openrouter,google/gemini-2.5-flash",
    "longContext": "openrouter,google/gemini-2.5-flash",
    "longContextThreshold": 60000,
    "webSearch": "openrouter,google/gemini-2.5-flash"
  }
}
```

## 汎用エージェントと垂直エージェント

*2025.08*

[Manus](https://manus.im)は汎用AIエージェントツールであると主張されていますが、おそらくそれほど上手く機能しないでしょう。

一つの理由は、非常に動作が遅く、不要な作業が多く、非効率であることです。もう一つの理由は、複雑な問題に遭遇したり、弱点に当たったりすると、タスクに失敗する可能性が高いことです。

垂直エージェントは非常に専門化されているため、非常に優れた機能を発揮します。それらは非常に特定のタスクに合わせて調整されています。何十ものデータベースがあり、Springのような100以上のWeb開発フレームワークがあります。VueやReactのような多数のWebフレームワークもあります。

[Dify](https://dify.ai)は、AIを使用してワークフローを接続することに焦点を当てており、ドラッグアンドドロップ方式でAIワークフローを定義します。情報、データ、プラットフォームを接続するために多くの作業を行う必要があります。

私もいくつかのシンプルなエージェントを構築しました。例えば、Pythonコードのリファクタリングエージェント、文法修正エージェント、バグ修正エージェント、エッセイ結合エージェントなどです。

コードは非常に柔軟です。したがって、Difyは可能なアイデアの空間のごく一部しかカバーしていません。

Manusはタスクを実行し、VNC方式を使用してコンピュータを表示することで、ユーザーにその動作を示します。

将来はこれら2つのアプローチに落ち着くと私は思います。

Manusの場合、タスクを実行するためにコードやテキストをアップロードする必要がありますが、これは便利ではありません。Difyの場合、MIT Scratchと同様に、ドラッグアンドドロップを使用してワークフローを構築する必要があります。

なぜScratchはPythonほど人気がないのでしょうか？それは、Pythonでできることが非常に多いのに対し、Scratchは教育目的のシンプルなプログラムに限定されているからです。

Difyにも同様の制限がある可能性があります。

Manusは多くの簡単なタスクを処理できます。しかし、 Manusの弱点に触れるような一部のタスク、特にそれらのタスクでは失敗します。

また、多くのプログラムやサービスは設定に時間がかかります。Manusのアプローチでは、このプロセスは遅いです。

プログラマーとして、私はPythonとAIを使用して垂直エージェントを構築しています。これは私にとって最も簡単なアプローチです。プロンプトやコンテキストを設定して、LLM APIからの比較的安定した出力を確保することもできます。

ManusとDifyもこれらのLLM APIで構築されています。彼らの利点は、すでに多くのツールやコードをすぐに使えるようにしていることです。

Twitterボットエージェントを構築したい場合、Difyを使用する方が、オープンソース技術で自分で構築するよりも便利かもしれません。

## AIコーディングツールに対するうるさいエンジニアの見解

*2025.08*

最近、私はClaude Codeの実行に成功したので、私のツール選定の旅を共有したいと思います。また、その過程でいくつかの[AIツールチップス](ai-tips-en.md)も集めました。

私はClaude Codeの採用がかなり遅かったです。

[Claude Code](https://www.anthropic.com/news/claude-3-5-sonnet)は2025年2月末頃にリリースされました。

最近まで試すことができませんでした。一つの理由は、Anthropic APIが必要で、それが中国のVisaカードをサポートしていないからです。

もう一つの理由は、[Claude Code Router](https://github.com/musistudio/claude-code-router)が利用可能になり、最近の試みが成功したことです。

常に称賛の声を聞いていました。2025年7月にGemini CLIを試しましたが、コードの修正に何度か失敗した後、放棄しました。

別のソフトウェアエージェントであるAiderも試しました。CursorはVSCodeベースのプラグインの多くが誤動作したため、約6ヶ月で使用を中止しました。さらに、CursorがVSCodeの上に構築されていることを考えると、それほど評価したくありません。VSCodeのCopilotプラグインは最近改善され、それほど遅れをとっていないため、私はそちらをより頻繁に使用することを好みます。

しかし、VSCodeはオープンソース技術であるElectron上に構築されています。どのチームまたは個人に功績を帰すべきかを判断するのは難しいです。多くの大企業やスタートアップがオープンソースプロジェクトから利益を得ていることを考えると、私は自分の予算と自分に最適なものに集中しなければなりません。功績を帰することについてあまり心配する必要はありません。手頃で効果的なツールを使用することを好みます。

Clineを短期間試しましたが、採用しませんでした。

OpenRouterを介してカスタマイズされたモデル、Grok 3ベータ版を使用してVSCodeでCopilotプラグインを使用しています。これはうまく機能しています。

Claude Codeが私の習慣を変えるとは思いませんが、うまく実行できるようになったので、あと数回試す忍耐力があれば、今後数週間でどのように感じるか見てみましょう。

私は10年のソフトウェアエンジニアリング経験を持つこだわり屋のユーザーです。ツールが実際に使用する上で素晴らしいものであることを望んでいます。ブランドに惑わされることはなく、日々の有用性だけを気にしています。

この投稿の文法を修正するためにClaude Codeを使用したところ、特定のシナリオでうまく機能することを発見しました。文法支援のためのAIには感謝していますが（この目的のためにLLM APIを呼び出すPythonスクリプトも書きました）、フラストレーションを感じるパターンに気づきました。最小限の修正を要求しても、ツールはレビューのために多数の文法提案を表面化し続けます。この手動検証プロセスは、自動化の意味を失わせます。妥協策として、私は現在、AIにエッセイ全体を処理させていますが、このアプローチでは、具体的な修正内容がわからないため、学習機会が制限されます。

最も印象的だったのは、Claude Codeが変更を表示する方法でした。git diffのように変更前と変更後を比較表示することで、編集内容のレビューがはるかに簡単になります。

一日後、Claudeを使っていくつかのコードも修正しました。しかし、CopilotプラグインとGrok 3ベータモデルを使い続けています。私にとってはシンプルで使いやすいからです。

数日間Claude Codeを使用した結果、非常に印象的だと言わざるを得ません。コードの修正方法が本当に気に入っています。

## DeepseekとMistralのAPI使用状況

*2025.01.25*

### DeepSeek

1ヶ月間で1500万トークンの使用料金は約23.5人民元でした。

これはある日の私の使用状況です:

| 種類              | トークン数  |
|-------------------|-----------|
| 入力 (キャッシュヒット)  | 946,816   |
| 入力 (キャッシュミス) | 2,753,752 |
| 出力            | 3,100,977 |

計算は以下の通りです:

0.94 * 0.1 + 2.75 * 1 + 3.10 * 2 = 11.83

したがって、タスクによって、トークンの使用量は入力（キャッシュミス）と出力に大きく依存します。

この結果は、予想されるコストと一致しています。

### Mistral

Mistralモデルの料金は以下の通りです:

| モデル                 | 入力 (100万トークンあたりUSD) | 出力 (100万トークンあたりUSD) |
|-----------------------|------------------------------|---------------------------------|
| `mistral-large-2411`  | 2                            | 6                               |
| `mistral-small-latest`| 0.2                          | 0.6                             |

ある日の私のMistralアカウントの使用状況は以下の通りでした (モデル: `mistral-large-2411`):

| 種類   | トークン数  | 費用 (USD) |
|--------|---------|------------|
| 合計  | 772,284 | 3.44       |
| 出力 | 474,855 | 2.85       |
| 入力  | 297,429 | 0.59       |

`mistral-small-2409`モデルの場合、合計使用量は1,022,407トークンでした。

これらのうち1/3が入力トークン、2/3が出力トークンと仮定すると:

入力トークンが340,802、出力トークンが681,605でした。

したがって、総コストは340,802 * 0.2 / 1,000,000 + 681,605 * 0.6 / 1,000,000 = 0.07 + 0.41 = 0.48 USDと計算されます。

Mistralコンソールによると、総使用コストは0.43 USDであり、これは私たちの計算とほぼ一致しています。
