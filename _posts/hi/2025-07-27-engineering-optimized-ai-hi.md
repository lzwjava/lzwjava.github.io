इस ब्लॉग में, मैंने अनुवाद, प्लेग्राउंड, फ्रंटमैटर रखरखाव और टेलीग्राम बॉट्स के लिए सहायता के लिए सौों स्क्रिप्ट का उपयोग किया है। मुझे लगता है कि यह विकास दृष्टिकोण AI के लिए अनुकूलित सॉफ्टवेयर इंजीनियरिंग का भविष्य का प्रतिनिधित्व कर सकता है।

मैं बहुत अधिक Python मॉड्यूल फंक्शन पर निर्भर नहीं करता, न ही मैं बड़े Java Spring प्रोजेक्ट की तरह कोड को संरचित करना चाहता हूँ।

मेरे करियर में मैंने कई सॉफ्टवेयर प्रोजेक्ट्स पर काम किया है। मैंने प्रभावशाली बैंकिंग आर्किटेक्चर, माइक्रोसर्विसेज, प्रभावी मल्टी-कंट्री डिज़ाइन जो डुप्लिकेशन को कम करते हैं, Spring के ऊपर बने मजबूत फाउंडेशनल फ्रेमवर्क, और सेंट्रलाइज्ड कॉन्फ़िगरेशन के साथ मजबूत गवर्नेंस देखे हैं।

हालांकि ये बैंकिंग आर्किटेक्चर प्रभावशाली हैं, अगर हम आज से शुरू करते, तो मैं LLM और AI के लिए अनुकूलित करना चाहूंगा। इसमें बेहतर कॉन्टेक्स्ट इंजीनियरिंग, सुधारी हुई सेपरेशन ऑफ कंसर्न, और मानव-केंद्रित डिज़ाइन के बजाय AI-फर्स्ट थिंकिंग को प्राथमिकता देना शामिल होगा। हालांकि Spring कई लेयर्स और अच्छी एब्स्ट्रैक्शन प्रदान करता है, लेकिन LLM और AI के लिए इसे नेविगेट करना मुश्किल हो सकता है।

मुझे लगता है कि हमें फ्लैटर स्ट्रक्चर के लिए प्रयास करना चाहिए, जो एक फ्लैट संगठन के समान है। इसका मतलब है कि केवल दो स्तरों का उपयोग करना: पहला स्तर दूसरे स्तर को कॉल करता है। एक फंक्शन में, यह बेहतर है कि वह सीधे 50 अन्य फंक्शन को कॉल करे, बजाय 50 नैस्टेड लेयर्स या स्टैक्स के। AI/LLM जटिल, नैस्टेड स्ट्रक्चर को समझने या अनुमान लगाने में संघर्ष करते हैं, लेकिन वे 100 से 200 लाइनों के छोटे फंक्शन को संभालने में उत्कृष्ट हैं। Python अन्य फाइलों से कॉल और इम्पोर्ट करने के लिए उपयुक्त है।

Python कोड Java से आसान होने का एक कारण यह है कि इसकी डिपेंडेंसी मैनेजमेंट सरल है। आपको बस `pip install` का उपयोग करके एक डिपेंडेंसी जोड़नी होती है। Maven के साथ, आपको POM XML फाइल में डिपेंडेंसी लिखनी होती है और फिर `mvn compile` चलाना होता है ताकि Maven डिपेंडेंसी डाउनलोड कर सके।

Python की सरलता का एक और कारण यह है कि इसका कोड बिना किसी परेशानी के सीधे चलाया जा सकता है।

हालांकि Java 11 से शुरू, `java` कमांड एकल-फाइल स्रोत कोड प्रोग्राम को बिना `javac` का उपयोग करके अलग से कॉम्पाइल किए सीधे चलाने में सक्षम है। हालांकि, अक्सर Java प्रोजेक्ट्स बड़े होते हैं, इसलिए आपको `mvn spring-boot:run` के साथ कुछ प्रॉपर्टी कॉन्फ़िगरेशन चलानी होती है।

तीसरा कारण यह है कि Python का मॉड्यूल डिज़ाइन सरल है; आप `from` और `import` का उपयोग करके आसानी से अन्य फाइलों से कोड इम्पोर्ट कर सकते हैं।

अब, कई AI चैटबॉट्स Python कोड को सीधे चैटबॉट विंडो में चलाने में सक्षम हैं, जैसे Grok।

जब 100 Java फाइलें, हर एक में लगभग 1000 लाइनें, सरल Python स्क्रिप्ट्स के साथ तुलना की जाती है, तो यह एक न्यायसंगत तुलना नहीं है। इस तरह के प्रोजेक्ट के लिए, मैं 1000 Python फाइलें, हर एक में लगभग 100 लाइनें, रखना पसंद करूंगा।

कोड के लाइन या एक फंक्शन को संपादित करने के लिए चुनना स्वीकार्य है। लेकिन आपको जानना होगा कि कहाँ से चुनना है। क्यों नहीं इस कार्य को AI द्वारा संभालने दिया जाए ताकि हमारी जिंदगी आसान हो जाए? इसलिए, हमें बस "सभी चुनें" का उपयोग करके सभी कोड को चुनना होगा और AI/LLM को संपादित करने के बारे में बताना होगा।

Python के लिए, `if __name__ == "__main__":` का उपयोग करके फाइल में फंक्शन चलाने और टेस्ट करने के लिए आसान है। अन्य Python फाइलों के लिए भी इस फाइल में फंक्शन को कॉल करने के लिए आसान है, बिना टेस्ट चलाए।

यह AI के लिए अनुकूलित कॉन्टेक्स्ट इंजीनियरिंग है। क्या हम इसे अन्य तरीकों से कर सकते हैं? AI/LLM ऑटो-रीग्रेसिव है। हालांकि, जब हम Copilot या Claude Code का उपयोग करते हैं, तो हमें नहीं पता होता है कि AI सॉफ्टवेयर एजेंट हमें कैसे मदद करता है। उन्हें इसके बारे में सोचना चाहिए, न कि हम।

क्या हम, यूज़र साइड पर, कोड को विशेष रूप से व्यवस्थित कर सकते हैं ताकि टोकन उपयोग कम हो? इस बिंदु के लिए, 1000 Python फाइलें, हर एक में 100 लाइनें, इस उद्देश्य के लिए अच्छी है। क्योंकि आप फंक्शन और कोड फाइलों को आसानी से सत्यापित कर सकते हैं, इससे पहले कि अन्य Python कोड उन्हें कॉल करे।

लेकिन एक समस्या यह है कि अगर आप कई कोड फाइलों को एक साथ बदलना चाहते हैं, तो यह आसान नहीं है। एक सरल तरीका यह है कि आप कोड को AI चैटबॉट्स में कॉपी करें और उन्हें बताएं कि इन फाइलों में कोड को कैसे संपादित करना है।

शायद हमें फंक्शन या लॉजिक को अलग करने के लिए लाइनों की संख्या का उपयोग करने की आवश्यकता नहीं है। लेकिन हमें छोटे फंक्शन में लॉजिक को अलग करना चाहिए। हम इसे प्राकृतिक रूप से लॉजिक के प्रकार के अनुसार अलग करके कर सकते हैं, जिससे वे छोटे दिखते हैं।

हम AI के लिए अनुकूलित सॉफ्टवेयर इंजीनियरिंग क्यों चाहते हैं? क्योंकि AI शक्तिशाली है, हमें सब कुछ AI के लिए अनुकूलित करना चाहिए और फिर AI को सॉफ्टवेयर इंजीनियरिंग में जितना हो सके उतना मदद करने दें।

यह केवल कोड के लिए ही संभव नहीं है, बल्कि किसी भी टेक्स्ट के लिए भी है। मान लीजिए हम बहुत चयनात्मक संपादक हैं; हम नहीं चाहते कि AI हमारे बड़े टेक्स्ट को एक बार में संपादित करे। हम पैराग्राफ द्वारा पैराग्राफ जांचना चाहते हैं। कोड के लिए, हम छोटे त्रुटियों या बग्स को सहन कर सकते हैं। टेक्स्ट के लिए, हम उन्हें सहन कर सकते हैं क्योंकि अधिकांश पाठक इतने चयनात्मक नहीं होते।

लेकिन कोड अलग है कि कभी-कभी, एक छोटी सी त्रुटि भी एक बड़े प्रोजेक्ट के पूरी तरह से खराब होने का कारण बन सकती है।

XML या YAML फाइलों के लिए, शायद हमें उन्हें उतना अलग नहीं करना पड़े क्योंकि वे पहले से ही अत्यधिक संरचित हैं।

और HTML फाइलों के लिए, हमें कुछ अलग करना चाहिए। बजाय हजारों JavaScript फाइलों को हजारों HTML फाइलों के साथ लिखने के, जिससे 1000 लाइनों को पार करना आसान हो जाता है, हमें JavaScript के लिए `import` का उपयोग करना चाहिए। JavaScript कोड के लिए, हम ऊपर बताए गए तरीकों का उपयोग करके अलग कर सकते हैं।

हम कोड को ऐसे संरचित करना चाहते हैं कि AI हमें आसानी से कोड जोड़ने, संपादित करने, हटाने और चलाने में मदद कर सके। यह शुरुआत है। कल की कल्पना कीजिए जब AI आसानी से सभी कोड को जनरेट या सुधार सके। दुनिया अत्यधिक डिजिटलाइज्ड होगी।

अपने आप को 100 बड़े सॉफ्टवेयर प्रोजेक्ट लिखते हुए देखिए और दूसरों के साथ कनेक्ट करने के लिए APIs प्रदान करते हुए। इसमें मेरा दैनिक एजेंडा शामिल है; मैं खुद को आजकल एक 1000-कार्मिक टेक कंपनी के समान हूँ। वे मेरे आवश्यकताओं के लिए अनुकूलित हैं, पैसे कमाने या मेरे फायदे के लिए पैसे खर्च करने के लिए। यह वास्तव में आश्चर्यजनक है।
