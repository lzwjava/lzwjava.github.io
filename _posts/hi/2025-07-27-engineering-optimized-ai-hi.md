इस ब्लॉग में, मैंने हजारों स्क्रिप्ट्स का उपयोग किया है जो अनुवाद, प्लेग्राउंड, फ्रंटमैटर रखरखाव और टेलिग्राम बॉट्स के लिए सहायक रहे हैं। मुझे लगता है कि यह विकास दृष्टिकोण भविष्य के सॉफ्टवेयर इंजीनियरिंग का प्रतिनिधित्व कर सकता है जो AI के लिए अनुकूलित है।

मैं Python मॉड्यूल फंक्शनलिटी पर बहुत निर्भर नहीं करता, न ही मैं कोड को बड़े Java Spring प्रोजेक्ट की तरह संरचित करना चाहता हूँ।

मैंने अपने पिछले टीम लीड को बताया कि हालांकि उनका बैंकिंग आर्किटेक्चर प्रभावशाली है, अगर आज से शुरू करते तो मैं LLM/AI के लिए अनुकूलित करने पर विचार करूँगा। इसमें बेहतर कॉन्टेक्स्ट इंजीनियरिंग, सुधारित कॉन्टेक्स्टों का अलगाव और AI-फर्स्ट थिंकिंग को मानव-केंद्रित डिजाइन पर प्राथमिकता देना शामिल होगा। जबकि Spring कई स्तरों और अच्छे एब्स्ट्रैक्शन प्रदान करता है, LLM/AI के लिए इसे नेविगेट करना चुनौतीपूर्ण हो सकता है।

मुझे लगता है कि हमें फ्लैटर संरचनाओं की ओर लक्ष्य रखना चाहिए, जैसे एक फ्लैटर संगठन। इसका मतलब है कि केवल दो स्तरों का उपयोग करना: पहला स्तर दूसरे स्तर को कॉल करता है। एक फंक्शन में, बेहतर है कि वह सीधे 50 फंक्शन को कॉल करे, बजाय 50 नैस्टेड स्तरों या स्टैक्स के। AI/LLM अत्यधिक जटिल, नैस्टेड संरचनाओं का मूल्यांकन या अनुमान लगाने में संघर्ष करते हैं, लेकिन वे 100 से 200 लाइनों के छोटे फंक्शन्स को संभालने में उत्कृष्ट हैं। Python अन्य फाइलों से कॉल और इम्पोर्ट करने के लिए उपयुक्त है।

Python कोड Java से आसान होने का एक कारण यह है कि इसका डिपेंडेंसी मैनेजमेंट सरल है। आपको बस `pip install` का उपयोग करके एक डिपेंडेंसी जोड़नी होती है। Maven के साथ, आपको POM XML फाइल में डिपेंडेंसी लिखनी होती है और फिर `mvn compile` चलाना होता है ताकि Maven डिपेंडेंसी डाउनलोड कर सके।

Python की सरलता का एक और कारण यह है कि इसका कोड बिना किसी परेशानी के सीधे चलाया जा सकता है।

हालांकि Java 11 से, `java` कमांड एकल-फाइल स्रोत-कोड प्रोग्राम को सीधे चलाने में सक्षम है बिना `javac` का उपयोग करके उन्हें अलग से कॉम्पाइल किए। हालांकि, अक्सर Java प्रोजेक्ट बड़े होते हैं, इसलिए आपको उन्हें `mvn spring-boot:run` के साथ कुछ प्रॉपर्टी कॉन्फ़िगरेशन के साथ चलाना पड़ता है।

तीसरा कारण यह है कि Python का मॉड्यूल डिजाइन सरल है; आप `from` और `import` का उपयोग करके आसानी से अन्य फाइलों से कोड इम्पोर्ट कर सकते हैं।

अब तक, कई AI चैटबॉट्स Python कोड को सीधे चैटबॉट विंडो में चलाने में सक्षम हैं, जैसे Grok।

100 Java फाइलों की तुलना, जिनमें से हर एक में लगभग 1000 लाइनें होती हैं, कुछ सरल Python स्क्रिप्ट्स के साथ, यह एक न्यायसंगत तुलना नहीं है। इस तरह के प्रोजेक्ट के लिए, मैं 1000 Python फाइलों को पसंद करूँगा, जिनमें से हर एक में लगभग 100 लाइनें होंगी।

कोड के लाइनों या एक फंक्शन को संपादित करने के लिए चुनना स्वीकार्य है। लेकिन आपको जानना चाहिए कि कहाँ से चुनना है। क्यों नहीं इस कार्य को AI द्वारा संभालने दें ताकि हमारी जिंदगी आसान हो जाए? इसलिए, हमें बस "सभी चुनें" का उपयोग करके सभी कोड को चुनना चाहिए और AI/LLM को बताना चाहिए कि इसे कैसे संपादित करना है।

Python के लिए, `if __name__ == "__main__":` का उपयोग करके फाइलों में फंक्शन चलाना और टेस्ट करना आसान है। अन्य Python फाइलों को इस फाइल में फंक्शन को कॉल करने के लिए भी आसान है बिना टेस्ट चलाने की आवश्यकता के।

यह AI के लिए अनुकूलित कॉन्टेक्स्ट इंजीनियरिंग है। क्या हम इसे अन्य तरीकों से कर सकते हैं? AI/LLM ऑटो-रीग्रेसिव है। हालांकि, जब हम Copilot या Claude Code का उपयोग करते हैं, तो हम नहीं जानते कि AI सॉफ्टवेयर एजेंट हमें कैसे मदद करता है। उन्हें इसके बारे में सोचना चाहिए, न कि हम।

क्या हम, यूजर साइड पर, कोड को विशेष रूप से व्यवस्थित कर सकते हैं ताकि टोकन उपयोग कम हो? इस बिंदु के लिए, 1000 Python फाइलों का दृष्टिकोण, जिनमें से हर एक में 100 लाइनें हैं, इस उद्देश्य के लिए अच्छा है। क्योंकि आप फंक्शन और कोड फाइलों को आसानी से सत्यापित कर सकते हैं पहले कि अन्य Python कोड उन्हें कॉल करे।

लेकिन एक समस्या यह है कि अगर आप कई कोड फाइलों को साथ में बदलना चाहते हैं, तो यह आसान नहीं है। एक सरल तरीका यह है कि आप कोड को AI चैटबॉट्स में कॉपी करें और उन्हें बताएं कि इन फाइलों में कोड को कैसे संपादित करना है।

शायद हमें फंक्शन या लॉजिक को अलग करने के लिए लाइनों की संख्या का उपयोग करने की आवश्यकता नहीं है। लेकिन हमें इसे छोटे फंक्शन में अलग करना चाहिए। हम इसे प्राकृतिक रूप से लॉजिक के प्रकार के अनुसार अलग करके कर सकते हैं, जिससे वे छोटे लगते हैं।

हम AI के लिए सॉफ्टवेयर इंजीनियरिंग क्यों चाहते हैं? क्योंकि AI शक्तिशाली है, हमें सब कुछ AI के लिए अनुकूलित करना चाहिए और फिर AI को सॉफ्टवेयर इंजीनियरिंग में जितना हो सके उतना मदद करने दें।

यह केवल कोड के लिए ही संभव नहीं है, बल्कि किसी भी टेक्स्ट के लिए भी है। मान लीजिए हम बहुत चयनात्मक संपादक हैं; हम नहीं चाहते कि AI हमारे बड़े टेक्स्ट को एक बार में संपादित करे। हम पैराग्राफ द्वारा पैराग्राफ सत्यापित करना चाहते हैं। कोड के लिए, हम छोटे त्रुटियों या बग्स को सहन कर सकते हैं। टेक्स्ट के लिए, हम उन्हें सहन कर सकते हैं क्योंकि अधिकांश पाठक इतने चयनात्मक नहीं होते।

लेकिन कोड अलग है कि कभी-कभी, एक छोटी सी त्रुटि भी एक बड़े प्रोजेक्ट के पूरी तरह से खराब होने का कारण बन सकती है।

XML या YAML फाइलों के लिए, शायद हमें उन्हें उतना अलग नहीं करना पड़े क्योंकि वे पहले से ही अत्यधिक संरचित हैं।

और HTML फाइलों के लिए, हमें कुछ अलगाव करना चाहिए। हजारों JavaScript फाइलों को हजारों HTML फाइलों के साथ लिखने के बजाय, जिससे 1000 लाइनों को पार करना आसान हो जाता है, हमें JavaScript के लिए `import` का उपयोग करना चाहिए। JavaScript कोड के लिए, हम ऊपर बताए गए तरीकों का उपयोग कर सकते हैं।

हम कोड को ऐसे ढंग से संरचित करना चाहते हैं कि AI हमें आसानी से कोड जोड़ने, संपादित करने, हटाने और चलाने में मदद कर सके। यह शुरुआत है। एक दिन की कल्पना करें जब AI आसानी से सभी कोड को उत्पन्न या ठीक कर सके। दुनिया अत्यधिक डिजिटल हो जाएगी।

मुझे खुद को 100 बड़े सॉफ्टवेयर प्रोजेक्ट लिखते हुए देखो और दूसरों के साथ कनेक्ट करने के लिए APIs प्रदान करता हूँ। इसमें मेरा दैनिक एजेंडा शामिल है; मैं खुद को आजकल एक 1000-कार्मिक टेक कंपनी की तरह मानता हूँ। वे मेरे आवश्यकताओं के अनुसार अनुकूलित हैं, पैसा कमाने या खर्च करने के लिए मेरे फायदे के लिए। यह वास्तव में आश्चर्यजनक है।
